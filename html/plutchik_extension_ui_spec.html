Shadow DOM badge injection isolation pattern ▾
function injectBadge(element, prediction) { const host = document.createElement('span') host.style.cssText = 'display:inline-block;margin-left:6px' // Closed shadow root — no style bleed in or out const shadow = host.attachShadow({ mode: 'closed' }) const badge = buildBadge(prediction) // prediction = { emotion, confidence, ring, // sarcasm_prob, hitl_flag, model_used } shadow.appendChild(badge) element.appendChild(host) } function buildBadge(p) { const el = document.createElement('span') if (p.hitl_flag) { el.textContent = 'low confidence' el.className = 'badge low-conf' } else { el.textContent = `${p.emotion} · ${Math.round(p.confidence * 100)}%` el.className = `badge ${p.ring}` } return el }
Adding a custom site adapter developer API ▾
Register any site by implementing four methods. Drop the file in plugin/adapters/ and it auto-registers.
// plugin/adapters/my_site_adapter.js export const MySiteAdapter = { name: 'mysite', // 1. URL match — when to activate match: (url) => url.includes('mysite.com'), // 2. What to scan — returns [{text, element}] extract: () => { return [...document.querySelectorAll('.message-body')] .map(el => ({ text: el.innerText.trim(), element: el })) .filter(m => m.text.length >= 30) }, // 3. Model hint — override auto-routing modelHint: 'nemotron', // or 'roberta' or 'auto' // 4. Where to attach the badge badgeTarget: (element) => element.querySelector('.message-footer') ?? element }
The modelHint field lets you force a model for a specific site. If 'auto', the routing engine decides based on context signals.
Inference pipeline — how a scan works message passing ▾
// content_script.js → background_worker.js // Step 1: content script extracts texts from DOM const adapter = AdapterRegistry.resolve(location.href) const items = adapter.extract() // Step 2: batch send to background worker chrome.runtime.sendMessage({ type: 'SCAN_BATCH', items: items.map(i => i.text), site: adapter.name, modelHint: adapter.modelHint, }) // background_worker.js // Step 3: route to correct model, run WASM inference chrome.runtime.onMessage.addListener((msg, _, reply) => { const model = ModelRouter.select(msg.site, msg.items) // model = 'roberta' | 'nemotron' const results = model.inferBatch(msg.items) // Step 4: wrap each result through HITL assessor const assessed = results.map(r => HITLAssessor.assess(r)) // Step 5: send back to content script for injection reply({ predictions: assessed }) })
The routing engine picks the model at runtime. It is never hardcoded — it reads five signals and decides.
Five routing signals how the decision is made ▾
class ModelRouter { static select(site, texts) { const signals = { // Signal 1: site identity // gmail + linkedin = professional register → Nemotron // reddit + generic = conversational → RoBERTa siteScore: SITE_MODEL_MAP[site] ?? 'roberta', // Signal 2: average text length // Short texts (<20 words) → RoBERTa (fast, sufficient) // Long texts (>80 words) → Nemotron (better long context) lengthScore: avgWords(texts) > 80 ? 'nemotron' : 'roberta', // Signal 3: formal register heuristic // Detects formal markers: "Dear", "Regards", "Per our call" formalScore: hasFormalMarkers(texts) ? 'nemotron' : 'roberta', // Signal 4: developer modelHint override // Adapter can force a model — takes priority hintScore: adapter.modelHint !== 'auto' ? adapter.modelHint : null, // Signal 5: device capability // Nemotron requires WebGPU. Fall back to RoBERTa if unavailable. capScore: navigator.gpu ? null : 'roberta', } // Priority: hint > capability > formal > length > site return signals.hintScore ?? signals.capScore ?? majority([signals.formalScore, signals.lengthScore, signals.siteScore]) } }
RoBERTa vs Nemotron — what each model is for capability matrix ▾
RoBERTa Fast · WASM · CPU
Best for short conversational text. Reddit comments, chat messages, short social posts. Runs entirely on CPU via WASM. 82M params, <300ms. The default model on any site where the adapter doesn't specify otherwise.
Triggers: reddit, generic, avg text <80 words, no GPU
Nemotron Deep · WebGPU · Context
Best for professional register, long-form text, or when dissonance between formal surface and emotional subtext is the primary signal. Gmail threads, LinkedIn posts, customer support tickets. Requires WebGPU. Better at detecting corporate sarcasm and passive contempt.
Triggers: gmail, linkedin, formal markers, avg text >80 words, GPU available
Both models run fully on-device. No text is sent to any server. Model files load lazily on first scan — RoBERTa loads in ~1.2s, Nemotron in ~3.8s. Both cache in memory for the session.

Privacy
All inference runs on-device. No text is sent to any external server at any point.
HITL contribution is opt-in. If enabled, low-confidence samples are sent to the annotator queue with text only — no URL, no username, no metadata.
The extension uses activeTab permission only. It cannot access any page you haven't opened in the current tab.
